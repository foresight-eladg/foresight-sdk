Introduction

1.1 Purpose of the SDK

The Foresight SDK provides developers and integrators with a robust and flexible software interface to access, control, and process data from Foresight's multi-spectral stereo vision systems. It enables the integration of advanced perception capabilities, such as obstacle detection and depth estimation, into automotive, agricultural, industrial, and autonomous platforms.

The SDK supports the development and evaluation of perception algorithms, data recording, and system-level integration.

1.2 Target Audience

This SDK is intended for:

  • Software developers and system integrators
  • R&D engineers and technical teams
  • Teams working in ADAS, autonomous vehicles, and robotics
  • Intelligent vision application developers
  • Test engineers and data analysts

It assumes familiarity with C++ and Linux environments, as well as a basic understanding of computer vision and sensor interfacing.

1.3 System Overview

The Foresight system combines stereo vision technology with multi-spectral imaging, using both visible light (VIS) and infrared (IR) sensors to deliver enhanced perception in a wide range of environmental conditions.

The SDK supports multiple camera configurations, including dual and quad setups, and enables synchronization across sensors over interfaces such as USB, Ethernet, and more.

The Foresight SDK utilizes a client-server model where the SDK (client) connects to the Foresight service (server) to send requests, receive data, and control system behavior. The provided SDK libraries and headers abstract the underlying transport protocols, making it easy to integrate into various client-side workflows.

1.4 Data Accessing API

The SDK provides high-level APIs for accessing:

  • Raw and rectified stereo image streams
  • Depth maps
  • Obstacle classification outputs
  • Drivable area
  • Point clouds
  • Ego vehicle data

1.5 Data Sending API

The SDK provides high-level APIs for sending data over UDP protocol to the Foresight service.

1.6 Communication Protocols

UDP

Used for client-server communication and high-bandwidth data transfer.

MQTT

Used for internal service communication and lightweight messaging.

1.7 Glossary

Term Definition

2.2 Callbacks

2.2.1 Overview

The Foresight SDK Receiver is structured around two core components: Callbacks and Sender.

The Callbacks component is responsible for handling real-time data received from the Foresight service.

The SDK provides a set of predefined callback interfaces that developers must override to process incoming data. This design allows flexible, application-specific handling of various data types, such as object detection results and frame metadata.

All user-defined data structures and types needed to implement these callbacks are provided in the header files located at:

/opt/foresight/include/foresight/sdk

2.2.2 Callbacks Definition

Callbacks in the Foresight SDK are defined as virtual functions within specific interface classes. Developers are expected to inherit from these interfaces and override the relevant callback methods to handle incoming data according to their application's needs.

Each callback corresponds to a specific data type or event, such as object detection or frame-level metadata. When new data is received from the Foresight service, the SDK automatically invokes the appropriate callback function, passing the relevant data structures as arguments.

This architecture enables seamless integration of the SDK's data stream into user applications, supporting customized data processing and real-time responsiveness.

Below is a list of all available callbacks:

The Callbacks class is organized into three categories:

  • Vision Service Callbacks — Handle core vision processing events: logging, termination, frame numbers, motion data, metadata paths, and heartbeat signals.
  • Platform Service Callbacks — Handle platform-level events: logging and heartbeat signals from the Platform Service.
  • RGUI Callbacks — Handle real-time output data: images, point clouds, masks, transforms, VIO, edge detection, obstacles, statistics, calibration status, and more.
C++ Callbacks Interface Class (Complete) 
/**

{

2.2.3 Data Request

The following callback manages the configuration of the data request process:

virtual std::map<std::string, foresight::BitMsgType> getRequestData() = 0;

The function must return a mapping of stream names for each data request of type BitMsgType.

C++ Example: Data Request Configuration 
std::map<std::string, foresight::BitMsgType> getRequestData() override
{
    std::map<std::string, foresight::BitMsgType> request_data;
    request_data["front_vis"] = foresight::BitMsgType::CAMERA_FRAME_PRIMARY |
                                foresight::BitMsgType::CAMERA_FRAME_SECONDARY |
                                foresight::BitMsgType::OBJ_TRACKER_CLASSIFIER;
    request_data["front_ir"] =  foresight::BitMsgType::CAMERA_FRAME_PRIMARY |
                                foresight::BitMsgType::CAMERA_FRAME_SECONDARY |
                                foresight::BitMsgType::OBJ_TRACKER_CLASSIFIER;
    return request_data;
}

The Sender object uses the callback mapping to deliver the requested data to the client. Once the Sender object is initialized, this mapping cannot be modified.

2.2.4 Buffer Data Parsing

The callback function holds references to the parsed data.

VisualInertialOdometryData, EdgeDetectionData, ObstaclesData and etc…

MatrixData
MatrixData containing a pointer to a memory buffer. getBuffer() function retrieves the pointer to the data.
The section below explains how to treat buffered data.

Image Buffer: Primary Image, Secondary Image

BitMsgTypes: CAMERA_FRAME_PRIMARY, CAMERA_FRAME_SECONDARY

The image buffer is a one-dimensional array with a size calculated as m_length = m_width × m_height × bytes_per_pixel. The bytes per pixel size can be calculated from the bits per pixel (BPP) table at the end of this document.

Different system modes can send different formats:

ModeStream TypeFormatBytes per Pixel
AlgoVISRGB (interleaved)3
AlgoIRGray81
RecordVISYUV422P (planar)2
RecordIRGray81

Each value is stored as a uint8_t.

Example:
VIS image of RGB format will appear as follows:

C++ Image Buffer Parsing (RGB) 
pixel0.r = m_buffer[0]
pixel0.g = m_buffer[1]
pixel0.b = m_buffer[2]
pixel1.r = m_buffer[3]
...

for (int row = 0; row < height; ++row) {
    for (int col = 0; col < width; ++col) {
        // Calculate the starting index for this pixel
        size_t pixelIndex = (row * width + col) * channels;

        for (int k = 0; k < channels; ++k) {
            uint8_t pixelValue = buffer[pixelIndex + k];
        }
    }
}
Point Cloud Buffer: Filtered PCL (Point Cloud)

BitMsgTypes: FILTERED_PCL

PCL buffer: m_length = m_width × m_height × 3 (X Y Z for every pixel). Each value is stored as a float32.

C++ Point Cloud Buffer Parsing 
for (int row = 0; row < height; ++row) {
    for (int col = 0; col < width; ++col) {
        size_t pixelIndex = (row * width + col) * channels;

        for (int k = 0; k < channels; ++k) {
            float pixelValue = buffer[pixelIndex + k];
        }
    }
}
Depth Map Buffer

BitMsgTypes: DEPTH_PCL

Depth PCL buffer: m_length = m_width × m_height. Every pixel is of float32 size.

C++ Depth Map Buffer Parsing 
for (int row = 0; row < height; ++row) {
    for (int col = 0; col < width; ++col) {
        size_t pixelIndex = (row * width + col);
        float pixelValue = buffer[pixelIndex];
    }
}
Mask Image Buffer: Drivable Area, Rail Path Mask

BitMsgTypes: DRIVABLE_AREA, RAIL_PATH_MASK

Mask image buffer: m_length = m_width × m_height. Every pixel is of uint8_t size.

Values: 0 = ignored area, 255 = active area

C++ Mask Image Buffer Parsing 
for (int row = 0; row < height; ++row) {
    for (int col = 0; col < width; ++col) {
        size_t pixelIndex = (row * width + col);
        uint8_t maskValue = buffer[pixelIndex];
        
        if (maskValue == 255) {
            // Active area (drivable / rail path)
        } else {
            // Ignored area
        }
    }
}

Buffer Size Calculation

The image buffer is a one-dimensional array with size: m_length = m_width × m_height × bytes_per_pixel

The bytes per pixel can be calculated from the Bits per pixel (BPP) table: bytes_per_pixel = bpp / 8

2.3 Client—Foresight Service Communication

While the Callbacks component handles incoming data from the Foresight service, the Sender component is responsible for sending requests to the FS service.

Foresight SDK Sender

This section is for clients who want to send data from their video sources to the Foresight service via UDP. Clients using USB, Ethernet, and GMSL cameras can ignore this section.

3.1 Overview

The Remote Camera (RC) system is responsible for providing a synchronized video stream from two cameras: one positioned on the left and the other on the right. These two image sources must be combined into a single synchronized frame before transmission.

3.3 Communication Protocols

3.3.1 Video Frames Transmitting Protocol

The video stream channel uses UDP protocol. The RC software performs the following steps:

  1. Captures two images from its left and right video sources.
  2. Combines them into a single synchronized frame.
  3. Uses the provided SDK interface to send the frame to the FS via UDP.

3.3.2 Data Chunking and Transmission

  • The SDK interface automatically splits each combined frame into smaller chunks to ensure no UDP datagram exceeds the MTU.
  • Each datagram includes a custom header and a data segment.
  • All chunks belonging to the same frame share the same sequence_number and have an incrementing segment_number.

The default UDP packet size used by Foresight is 1400 bytes, including the header. This value can be modified upon mutual agreement.

Frame Length Calculation

The total frame length in bytes can be calculated using:

frame_length = (width × height × bpp) / 8

Where: width = frame width (e.g., 3840), height = frame height (e.g., 1232), bpp = bits per pixel

3.4.3 Execution Steps

  1. Parse command-line arguments: retrieve IP address, UDP port, and packet size.
  2. Create ClientUdpSender instance: instantiate using parsed IP, port, and packet size.
  3. Initialize the UDP socket: call connect_to_remote().
  4. Calculate frame length: frame_length = width × height × bpp / 8
  5. Read video frames: capture left and right images.
  6. Send frames in a loop: use sendFrames() method.
  7. Close UDP socket: call close() method.

    std::string ip_address = "127.0.0.1";
    uint32_t port = 31314;
    size_t packet_size = 1400;
    
    // Frame dimensions
    int width = 1920;
    int height = 1080;
    int bpp = 24; // RGB format
    size_t frame_length = (width * height * bpp) / 8;
    
    // Create sender instance
    foresight::ClientUdpSender sender(ip_address, port, packet_size);
    
    // Initialize connection
    auto error = sender.connect_to_remote();
    if (error) {
        std::cerr << "Failed to connect: " << error.message() << std::endl;
        return -1;
    }
    
    // Allocate frame buffers
    std::vector<uint8_t> left_frame(frame_length);
    std::vector<uint8_t> right_frame(frame_length);
    
    // Main loop - send frames
    while (running) {
        // Capture/generate frame data...
        
        // Send stereo pair
        sender.sendFrames(left_frame.data(), right_frame.data(), frame_length);
    }
    
    // Cleanup
    sender.close();
    return 0;
}

Resource Management
